Ćwiczenia 11 – Dokumentacja projektu

Wykonaj poniższe zadanie w kontekście własnego projektu bazy danych przygotowywanego w środowisku SQL Server Management Studio. Celem zajęć jest przygotowanie dokumentacji technicznej projektu, który był rozwiajny na poprzednich ćwiczeniach.

Dokumentacja projektu nie powinna być wyłącznie opisem „co robi system”. W przypadku projektu bazodanowego powinna wyjaśniać, jaki problem rozwiązuje baza, jakie dane są przechowywane, jak zaprojektowano tabele, relacje, klucze, ograniczenia, zapytania, procedury oraz elementy serwerowe. Dobra dokumentacja pozwala innej osobie zrozumieć projekt, odtworzyć bazę, sprawdzić poprawność rozwiązań i dalej rozwijać system.

Zakres projektu

  1. Skrypt odtwarzający bazę:
    • tworzenie struktury bazy danych,
    • wypełnienie bazy danymi testowymi,
    • możliwość uruchomienia skryptu na pustej instancji SQL Server.
  2. Zapytania DQL i elementy T-SQL:
    • SELECT,
    • JOIN,
    • agregacje,
    • podzapytania,
    • wybrane elementy T-SQL.
  3. Procedury składowane i funkcje:
    • procedury realizujące operacje biznesowe,
    • funkcje pomocnicze,
    • walidacje,
    • podsumowania i raporty.
  4. Transakcje:
    • BEGIN TRAN,
    • COMMIT,
    • ROLLBACK,
    • opis sytuacji, w których transakcja chroni spójność danych.
  5. Wybrane zagadnienia serwerowe:
    • uprawnienia i role,
    • widoki,
    • triggery,
    • logowanie zmian,
    • inne mechanizmy uzasadnione tematyką projektu.

Zawartość dokumentacji projektu

  1. Stronę tytułową lub krótką metrykę projektu:
    • nazwa projektu,
    • autor,
    • krótki opis dziedziny problemowej,
    • informacja o wykorzystanym systemie bazodanowym.
  2. Cel i zakres projektu:
    • jaki problem rozwiązuje baza danych,
    • kto będzie korzystał z danych,
    • jakie procesy biznesowe mają być obsługiwane,
    • czego projekt nie obejmuje.
  3. Opis modelu danych:
    • diagram ERD,
    • opis encji,
    • opis relacji i kardynalności,
    • uzasadnienie najważniejszych decyzji projektowych.
  4. Słownik danych:
    • nazwa tabeli,
    • nazwa kolumny,
    • typ danych,
    • klucz główny lub obcy,
    • wymagalność wartości,
    • ograniczenia i reguły walidacji,
    • krótki opis znaczenia pola.
  5. Opis implementacji:
    • najważniejsze fragmenty DDL,
    • więzy integralności,
    • indeksy,
    • przykładowe dane testowe,
    • kolejność uruchamiania skryptów.
  6. Opis zapytań, procedur i funkcji:
    • cel zapytania lub procedury,
    • parametry wejściowe,
    • wynik działania,
    • przykład użycia.
  7. Opis mechanizmów bezpieczeństwa i administracji:
    • role użytkowników,
    • uprawnienia,
    • widoki ograniczające dostęp do danych,
    • triggery i automatyzacja,
    • logowanie zmian.

Zadanie do wykonania na dzisiejszych ćwiczeniach:

  1. Przygotuj pierwszą wersję dokumentacji własnego projektu bazy danych.
  2. Dokumentacja może mieć formę:
    • pliku PDF,
    • strony HTML,
    • repozytorium GitHub z plikiem README.md,
    • strony GitHub Pages,
    • innej czytelnej formy uzgodnionej z prowadzącym.
  3. W pierwszej wersji dokumentacji uwzględnij:
    • nazwę projektu,
    • cel i zakres projektu,
    • opis dziedziny problemowej,
    • opis użytkowników lub procesów, które baza ma obsługiwać,
    • diagram ERD,
    • opis encji i relacji,
    • wstępny słownik danych,
    • krótkie uzasadnienie najważniejszych decyzji projektowych.
  4. Dokumentacja powinna być przygotowana tak, aby można było ją później uzupełnić o:
    • skrypt DDL,
    • skrypt DML z danymi testowymi,
    • zapytania DQL,
    • procedury i funkcje,
    • transakcje,
    • role, widoki, triggery i mechanizmy logowania zmian.

Do oddania

  1. Dokumentacja projektu w wybranej formie: .pdf, .html, .md albo link do opublikowanej strony.
  2. Diagram ERD jako część dokumentacji lub jako osobny załącznik.

Przykłady dokumentacji projektowej

Poniższe przykłady pokazują różne sposoby przygotowania dokumentacji: klasyczny dokument PDF, dokumentację architektury, stronę GitHub Pages oraz zbiór zasad dokumentowania projektu. Nie należy kopiować ich układu bezrefleksyjnie, ale warto przeanalizować, jak porządkują informacje o celu, zakresie, architekturze, komponentach i decyzjach technicznych.

  1. Red Alert – Software Design Doc
    https://ceias.nau.edu/capstone/projects/CS/2022/RedAlert/assets/documents/Software%20Design%20Doc.pdf

    Dokumentacja projektu aplikacji webowej. Przykład jest przydatny, ponieważ pokazuje opis technologii oraz diagram architektury systemu. W dokumencie wskazano m.in. wykorzystanie Django, MongoDB, JavaScriptu, NGINX i Gunicorn, a następnie opisano, jak te elementy współpracują w aplikacji.

  2. WiChat – dokumentacja projektu w szablonie arc42 / GitHub Pages
    https://arquisoft.github.io/wichat_en1a/ GitHub Pages Quickstart

    Jest to dokumentacja projektu opublikowana jako strona internetowa. Wykorzystuje układ dokumentacji architektury systemu oparty na arc42. Dokumentacja opisuje m.in. wymagania, ograniczenia, kontekst systemu, komunikację z zewnętrznymi usługami, widok komponentów i elementy jakościowe.

    arc42 to szablon dokumentowania architektury oprogramowania i systemów informatycznych. Nie jest to konkretne narzędzie, lecz uporządkowany standard opisu projektu, który podpowiada, jakie elementy warto udokumentować: cele systemu, wymagania jakościowe, ograniczenia, kontekst biznesowy i techniczny, strategię rozwiązania, podział na komponenty, widok uruchomieniowy, widok wdrożeniowy, decyzje architektoniczne, ryzyka oraz słownik pojęć. Oficjalny przewodnik: arc42 Documentation Guide.

  3. Microsoft Engineering Fundamentals Playbook – Documentation
    https://microsoft.github.io/code-with-engineering-playbook/documentation/

    Zbiór zasad dotyczących tworzenia dokumentacji w projektach informatycznych:

    • ogólne zasady dokumentowania projektów,
    • wyjaśnienie, dlaczego dokumentacja jest potrzebna także w projektach prowadzonych zwinnie,
    • wskazówki dotyczące dokumentacji architektury,
    • wskazówki dotyczące dokumentacji kodu, wdrożenia i decyzji projektowych,
    • przykładowe obszary, które warto dokumentować w projekcie.

Przykłady dokumentacji relacyjnych baz danych

  1. Music Store – Documentation to the Database Project
    https://www.science.uva.nl/COLNET/teaching/2007-dbp-ik/Projectmaterial/Team6_final.pdf

    Jest to dokumentacja studenckiego projektu bazy danych „Music Store”.

  2. Database Project – dokumentacja projektu bazy danych o akcjach giełdowych
    https://zanwarfaraj.github.io/database/

    Jest to przykład dokumentacji projektu bazodanowego opublikowanej jako strona internetowa. Autor opisuje projekt bazy MySQL hostowanej na AWS, służącej do przechowywania danych o akcjach giełdowych, oraz skrypt w Pythonie służący do czyszczenia i przenoszenia danych do bazy.

  3. Database Design Document – szablon dokumentacji bazy danych
    https://www.sdlcforms.com/PDFClientsDownload/Database_Design_Document.pdf

    Jest to szablon dokumentu projektowego dla bazy danych. Nie jest to gotowy projekt konkretnej aplikacji, ale przykład struktury, którą można wykorzystać przy tworzeniu własnej dokumentacji.
