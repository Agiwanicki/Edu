Ćwiczenia 8 – Transakcje, XML i JSON w SQL Server

Wykonaj poniższe przykłady w środowisku SQL Server Management Studio na bazie danych Northwind.

Celem ćwiczenia jest poznanie praktycznego wykorzystania transakcji w języku T-SQL, w tym kontroli liczby aktywnych transakcji, stosowania punktów zapisu, obsługi błędów oraz obserwowania blokowania danych przy pracy w dwóch sesjach. W drugiej części ćwiczenia pokazano również generowanie danych w formacie XML i JSON oraz podstawową analizę dokumentu JSON.

Transakcje w T-SQL

W ćwiczeniu wykorzystamy następujące instrukcje i funkcje:

  • BEGIN TRAN – rozpoczyna transakcję,
  • COMMIT – zatwierdza wykonane operacje,
  • ROLLBACK – wycofuje operacje wykonane w ramach transakcji,
  • SAVE TRAN – tworzy punkt zapisu, do którego można częściowo wycofać transakcję,
  • @@TRANCOUNT – zwraca liczbę aktywnych poziomów transakcji,
  • XACT_STATE() – pozwala sprawdzić, czy transakcja jest aktywna i czy można ją zatwierdzić,
  • TRY...CATCH – umożliwia przechwycenie i obsłużenie błędu.

Przykład 1 – zagnieżdżona transakcja z kontrolą @@TRANCOUNT i punktem zapisu

Cel: sprawdzenie, jak SQL Server interpretuje zagnieżdżone transakcje oraz jak działa punkt zapisu utworzony za pomocą SAVE TRAN.

W przykładzie rozpoczynamy transakcję zewnętrzną, dodajemy rekord do tabeli Shippers, tworzymy punkt zapisu, a następnie rozpoczynamy transakcję wewnętrzną. Po wykonaniu ROLLBACK TRAN Punkt1 wycofane zostaną zmiany wykonane po punkcie zapisu, natomiast sama liczba aktywnych transakcji nadal wymaga świadomego zakończenia instrukcją COMMIT lub pełnym ROLLBACK. 

BEGIN TRAN;
PRINT 'Po BEGIN TRAN: ' + CAST(@@TRANCOUNT AS VARCHAR);

INSERT INTO Shippers (CompanyName, Phone)
VALUES ('Test External', '111-111-111');

SAVE TRAN Punkt1;

BEGIN TRAN;
PRINT 'Po BEGIN transakcji wewnętrznej: ' + CAST(@@TRANCOUNT AS VARCHAR);

INSERT INTO Shippers (CompanyName, Phone)
VALUES ('Test Internal', '222-222-222');

ROLLBACK TRAN Punkt1;
PRINT 'Po ROLLBACK TRAN Punkt1: ' + CAST(@@TRANCOUNT AS VARCHAR);

SELECT *
FROM Shippers
WHERE CompanyName LIKE 'Test%';

COMMIT;
PRINT 'Po COMMIT: ' + CAST(@@TRANCOUNT AS VARCHAR);

Uwaga: punkt zapisu nie kończy transakcji. Pozwala jedynie cofnąć jej fragment. Dlatego po użyciu ROLLBACK TRAN Punkt1 transakcja nadal pozostaje aktywna.

Przykład 2 – transakcja z obsługą błędów i logowaniem błędu

Cel: przećwiczenie obsługi błędu w transakcji oraz zapisanie informacji o błędzie do tabeli pomocniczej.

W tym przykładzie celowo próbujemy dodać zamówienie dla nieistniejącego klienta. Naruszenie klucza obcego powinno spowodować błąd, który zostanie przechwycony w bloku CATCH. Następnie transakcja zostanie wycofana, a treść błędu zapisana w tabeli ErrorLog. 

IF OBJECT_ID('dbo.ErrorLog') IS NULL
BEGIN
    CREATE TABLE dbo.ErrorLog (
        ErrorMsg NVARCHAR(MAX),
        TranCount INT,
        LogTime DATETIME
    );
END;

BEGIN TRY
    BEGIN TRAN;
    PRINT 'START TRAN: ' + CAST(@@TRANCOUNT AS VARCHAR);

    INSERT INTO Orders (CustomerID, EmployeeID, OrderDate)
    VALUES ('XXX01', 1, GETDATE());

    COMMIT;
    PRINT 'COMMIT OK';
END TRY
BEGIN CATCH
    DECLARE @ErrMsg NVARCHAR(MAX) = ERROR_MESSAGE();
    DECLARE @ErrTranCount INT = @@TRANCOUNT;

    IF XACT_STATE() <> 0
        ROLLBACK;

    INSERT INTO dbo.ErrorLog (ErrorMsg, TranCount, LogTime)
    VALUES (@ErrMsg, @ErrTranCount, GETDATE());
END CATCH;

SELECT * FROM dbo.ErrorLog;

Warto zwrócić uwagę: najpierw zapamiętujemy treść błędu i wartość @@TRANCOUNT, a dopiero potem wykonujemy ROLLBACK. Dzięki temu w logu można sprawdzić, w jakim stanie znajdowała się transakcja w momencie wystąpienia problemu.

Przykład 3 – test współbieżności i blokowanie rekordu w dwóch sesjach

Cel: zaobserwowanie, że transakcja może zablokować rekord dla innych użytkowników do czasu jej zatwierdzenia lub wycofania.

Ten przykład należy wykonać w dwóch osobnych oknach zapytań w SSMS. Pierwsza sesja zakłada blokadę wyłączną na wybrany produkt. Druga sesja próbuje zmodyfikować ten sam rekord i czeka, aż pierwsza sesja zakończy transakcję.

Sesja 1 – pierwsze okno zapytań

BEGIN TRAN;

SELECT *
FROM Products WITH (XLOCK, ROWLOCK)
WHERE ProductID = 1;

-- Rekord jest zablokowany do momentu wykonania COMMIT lub ROLLBACK.
-- Nie zamykaj jeszcze tej transakcji. Przejdź do drugiego okna zapytań.

Sesja 2 – drugie okno zapytań

UPDATE Products
SET UnitPrice = UnitPrice + 1
WHERE ProductID = 1;

Powrót do sesji 1

COMMIT;

Po wykonaniu COMMIT w pierwszej sesji zapytanie z drugiej sesji powinno zostać odblokowane i zakończyć działanie.

Przykład 4 – transakcja wieloetapowa z wycofaniem wszystkich operacji

Cel: pokazanie, że wiele zależnych operacji może zostać potraktowanych jako jedna całość. Jeżeli jeden z etapów zakończy się błędem, wszystkie wcześniejsze operacje zostaną wycofane.

W przykładzie dodawany jest klient, zamówienie oraz pozycja zamówienia. Następnie celowo generowany jest błąd dzielenia przez zero. Po błędzie cała transakcja zostaje wycofana, dlatego klient i zamówienie nie powinny pozostać w bazie danych. 

BEGIN TRY
    BEGIN TRAN;

    INSERT INTO Customers (CustomerID, CompanyName)
    VALUES ('ZAA01', 'Zaawansowany Klient');

    DECLARE @OrderID INT;

    INSERT INTO Orders (CustomerID, EmployeeID, OrderDate)
    VALUES ('ZAA01', 1, GETDATE());

    SET @OrderID = SCOPE_IDENTITY();

    INSERT INTO [Order Details] (OrderID, ProductID, UnitPrice, Quantity, Discount)
    VALUES (@OrderID, 1, 10.0, 2, 0);

    DECLARE @Zero INT = 0;
    DECLARE @Test INT = 1 / @Zero;

    COMMIT;
END TRY
BEGIN CATCH
    IF XACT_STATE() <> 0
        ROLLBACK;

    PRINT 'Błąd: ' + ERROR_MESSAGE();
END CATCH;

SELECT * FROM Customers WHERE CustomerID = 'ZAA01';
SELECT * FROM Orders WHERE CustomerID = 'ZAA01';

Przykład 5 – procedura składowana sterująca COMMIT lub ROLLBACK

Cel: pokazanie, że logika transakcyjna może zostać zamknięta w procedurze składowanej, a decyzja o zatwierdzeniu lub wycofaniu zmian może zależeć od parametru.

Procedura dodaje testowego przewoźnika do tabeli Shippers. Jeżeli parametr ma wartość COMMIT, zmiana zostaje zatwierdzona. W przeciwnym razie transakcja zostaje wycofana. 

CREATE OR ALTER PROCEDURE dbo.ProceduraTransakcyjna
    @akcja VARCHAR(10)
AS
BEGIN
    BEGIN TRY
        BEGIN TRAN;

        INSERT INTO Shippers (CompanyName, Phone)
        VALUES ('Test Procedura', '123-456-789');

        IF UPPER(@akcja) = 'COMMIT'
            COMMIT;
        ELSE
            ROLLBACK;
    END TRY
    BEGIN CATCH
        IF XACT_STATE() <> 0
            ROLLBACK;

        PRINT 'Błąd: ' + ERROR_MESSAGE();
    END CATCH;
END;
GO

EXEC dbo.ProceduraTransakcyjna @akcja = 'COMMIT';
EXEC dbo.ProceduraTransakcyjna @akcja = 'ROLLBACK';

SELECT *
FROM Shippers
WHERE CompanyName = 'Test Procedura';

XML i JSON w SQL Server

W drugiej części ćwiczenia wykorzystamy mechanizmy SQL Server pozwalające wygenerować dane w formatach często używanych przy wymianie danych między systemami informatycznymi.

  • FOR XML – generuje wynik zapytania w formacie XML,
  • FOR JSON – generuje wynik zapytania w formacie JSON,
  • ISJSON() – sprawdza, czy tekst jest poprawnym dokumentem JSON,
  • OPENJSON() – pozwala odczytywać dane z dokumentu JSON,
  • JSON_VALUE() – pobiera pojedynczą wartość skalarną ze wskazanej ścieżki JSON.

Przykład 6 – generowanie dokumentu XML na podstawie zamówień

Cel: wygenerowanie dokumentu XML zawierającego zamówienia i ich pozycje.

Zapytanie pobiera zamówienia z ostatnich pięciu dni oraz powiązane z nimi pozycje zamówień. Następnie wynik zostaje przekształcony do dokumentu XML za pomocą FOR XML AUTO. 

SELECT 
    o.OrderID,
    o.OrderDate,
    o.ShipName,
    od.ProductID,
    od.Quantity,
    od.UnitPrice
FROM Orders o
JOIN [Order Details] od ON o.OrderID = od.OrderID
WHERE o.OrderDate >= DATEADD(DAY, -5, GETDATE())
FOR XML AUTO, ELEMENTS, ROOT('OrdersXML');

FOR XML AUTO tworzy strukturę XML automatycznie na podstawie aliasów tabel. Opcja ELEMENTS powoduje zapis danych jako elementów, a ROOT('OrdersXML') dodaje główny węzeł dokumentu.

Przykład 7 – generowanie i analiza danych JSON

Cel: wygenerowanie danych klientów w formacie JSON, zapisanie wyniku do zmiennej oraz odczytanie wybranych wartości z dokumentu JSON.

W przykładzie pobieramy klientów z Niemiec, generujemy dokument JSON, sprawdzamy jego poprawność funkcją ISJSON(), a następnie odczytujemy nazwy firm z tablicy Customers za pomocą OPENJSON(). 

DECLARE @json NVARCHAR(MAX);

SELECT CustomerID, CompanyName, City
FROM Customers
WHERE Country = 'Germany'
FOR JSON AUTO, ROOT('Customers');

SET @json = (
    SELECT CustomerID, CompanyName, City
    FROM Customers
    WHERE Country = 'Germany'
    FOR JSON AUTO, ROOT('Customers')
);

SELECT ISJSON(@json) AS IsValidJSON;

SELECT CompanyName
FROM OPENJSON(@json, '$.Customers')
WITH (
    CompanyName NVARCHAR(100) '$.CompanyName'
);

Uwaga: funkcja JSON_VALUE() jest wygodna, gdy chcemy odczytać pojedynczą wartość, np. nazwę pierwszej firmy z tablicy: 

SELECT JSON_VALUE(@json, '$.Customers[0].CompanyName') AS FirstCompanyName;
